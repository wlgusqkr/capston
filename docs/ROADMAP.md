# 로드맵 — 한계와 고도화

현재 시점에서 알려진 한계와 향후 개선 방향. 우선순위는 마감(2026-06-05) 기준.

---

## 1. 현재 한계

### 1.1 데이터
- **단독·다가구 좌표 누락**: 국토부 API 응답에 `jibun` 필드 자체가 없음 → `jibun=''` + `geom=NULL`로 적재. 17,339건 (전체 14.1%) 해당. 처리: 지도 핀에서는 제외(`geom__isnull=False`), 동 평균/시세 차트/점수에는 포함 (가격 데이터로서 유효)
- **단독·다가구 동 매핑 부정확**: 좌표가 없으니 `umdNm`(법정동명) + 구로 fallback인데, "행촌동·필운동" 같은 일부 법정동은 어떤 행정동에도 같은 이름이 없어 매칭 실패. 결과: 매칭된 거래도 한쪽 동으로 쏠림 (예: 관악구 단독·다가구 4,318건이 단 2개 동에만 매핑). 해결: 행안부 BJD↔ADM 매핑 CSV 적재 후 코드 기반 매칭
- **거래량 3건 미만 동**: 같은 구 평균 → 서울 중위 fallback. 425개 중 13개 (3%)
- **안전 / 자취생 비율 / 통학 시간 / 자취생 평점**: SPEC에 정의되어 있으나 데이터 미적재. 비교/상세에서 정직하게 표시 제외

### 1.2 기능
- **3D 시각화 미구현**: SPEC 우선순위 10번 (시간 남으면 deck.gl)
- **카카오 로그인 비활성**: 의존성은 추가되어 있으나 명시적 결정으로 비활성. SPEC 9단계 결정사항
- **자취생 리뷰 미구현**: 인증·모더레이션 비용 큼. `/api/users/me/reviews` 빈 리스트 반환

### 1.3 인프라
- **HTTPS 미적용**: 도메인 등록 + Let's Encrypt 미진행. IP 직접 접속 (`http://34.47.101.188`)
- **모바일 반응형 미적용**: viewport=1280 고정. 데모 PC 환경 가정
- **데이터 갱신 자동화 안 됨**: 현재 수동 스크립트. 월 1회 cron 또는 GitHub Actions 권장

## 2. 단기 (마감 전)

작업 우선순위 높음. 발표까지 가능한 범위.

| 항목 | 비용 | 가치 |
|---|---|---|
| HTTPS (Let's Encrypt) | 30분 — 도메인 + certbot | 발표 신뢰도 ↑ |
| Redis 캐시 hit rate 측정 | 10분 — 로깅 추가 | 성능 어필 근거 |
| 데이터 갱신 cron | 30분 — GitHub Actions 스케줄 | 운영 신뢰도 |
| Swagger UI 한국어 라벨 점검 | 20분 | 발표 시 시연용 |
| 모바일 viewport 응급 처리 | 1시간 — 최소 깨지지 않게 | 시연 PC 외 환경 |

## 3. 중기 (시간 남으면)

스코프는 SPEC 우선순위 9~10번.

### 3.1 3D 시각화 (deck.gl)
- 점수를 높이로 표현 (HexagonLayer 또는 ColumnLayer)
- 2D ↔ 3D 토글 (현재 ViewToggle 컴포넌트는 자리만 있음)

### 3.2 임베딩 기반 비슷한 동네
- 현재: score 벡터 거리 (Euclidean)
- 개선: 동별 amenity·transit·rent feature 벡터를 학습 임베딩으로 변환 (예: 동 사용자 행동 word2vec)

### 3.3 안전 지수
- 경찰청 범죄통계 (자치구 단위) → 행정동 단위 보간
- 서울 CCTV 위치 (서울 열린데이터광장) → 동별 카운트
- 둘을 결합한 `score_safety` 0~100

### 3.4 자취생 비율
- KOSIS 또는 통계청 1인 가구 비율 (행정동 단위) — 실제 자취생 비율과 차이 있으나 가장 가까운 프록시
- 동별 자취생 비율 → `single_household_pct` 필드

### 3.5 통학 시간
- 동 centroid → 주요 대학 (서울대·연세대·고려대·홍익대·이화여대 등) 카카오모빌리티/네이버 디렉션 API
- 사전 계산 (동 × 대학 매트릭스) — 실시간 호출 X

### 3.6 단독·다가구 동 매핑 정확도 개선
- 현재: `umdNm`(법정동명) 텍스트 일치로 fallback → 행정동에 같은 이름 없으면 skip 또는 한 동에 쏠림
- 개선: 행안부 `법정동코드 ↔ 행정동코드` 매핑 CSV 적재. 한 법정동 → 여러 행정동 분배는 비율(%)로 가중 적재 또는 1:1 대표 매칭
- `backend/scripts/build_dong_mapping.py` 가 골격만 있고 본문 미작성. CSV 받으면 즉시 동작
- 효과: 단독·다가구 17,339건이 더 균등하게 425개 동에 분포 → 개별 동 점수 정확도 ↑ (특히 관악·동대문·강북)

## 4. 장기 (캡스톤 이후)

상용 수준에서 고려할 만한 것.

### 4.1 데이터
- **갱신 주기 단축**: 현재 월 1회 → 주 1회 (실거래 갱신 따라가기)
- **다른 도시 확장**: 부산·대구 등. SPEC 14.2의 행정동 GeoJSON 교체만으로 가능
- **매물 이미지 / 시세 외부 연동**: 다방·직방 공개 API 또는 크롤링 (법적 검토 필요)

### 4.2 인프라
- **PostgreSQL 읽기 복제**: 동시 접속 1000+ 시
- **CDN**: 정적 자산 (CloudFront)
- **모니터링**: Sentry (에러), Grafana (메트릭)
- **API 레이트 리밋**: 현재 무제한. DRF throttle 활성화

### 4.3 기능
- **카카오 로그인 활성화**: 학부 데모용 의도적 미사용. 상용 시 필수
- **자취생 리뷰 시스템**: 인증 + 신고 + 모더레이션 + 어뷰징 방지
- **알림**: 즐겨찾기한 동의 평균 월세 변동 푸시
- **다국어**: 외국인 유학생 대상 EN/中/JP

### 4.4 ML
- **선호 학습 개선**: 현재 5쌍 → 더 많은 비교로 정확도 ↑. 능동학습(active learning)으로 정보량 최대화
- **개인화 추천**: 사용자 가중치 + 즐겨찾기 패턴 → "당신을 위한 동네 5개"
- **이상 탐지**: 거래 데이터에서 사기/허위 매물 탐지

## 5. 의도적 미선택

발표·심사 시 "왜 안 했나요?" 대비. 모두 명시적 결정.

| 항목 | 안 한 이유 |
|---|---|
| 마이크로서비스 분리 | 학부 캡스톤 트래픽 0~100명. CLAUDE.md 명시 결정 |
| Kafka / Elasticsearch | 단일 DB로 충분. 로그 검색은 journalctl |
| 회원 자체 가입 (이메일 인증 등) | username/password로 충분. 외부 SMTP 의존성 회피 |
| 매물 단위 정밀 좌표 | 지번 단위만 사용 (SPEC 14.2). 같은 지번 매물은 같은 좌표 |
| 매물 단위 지오코딩 | 위와 동일. 지번 단위 사전 매핑만 |
| 카카오 로그인 활성 | 도메인 등록·CSRF·비용 대비 학부 일정 부족. SPEC 9단계 결정 |
| 모바일 디자인 | 발표 환경 데스크톱 1280 고정. 시간 한정 |
| 서비스 워커 / PWA | 발표 데모 단발성 사용 |

## 6. 의사결정 기록 (SPEC 14)

명세서 14장 "주의사항 / 결정사항" 참고. 주요 항목:

- **5분 캐싱**: API 응답 5분 TTL Redis 캐시 (현재 활성)
- **이상치 처리**: IQR 1.5배 클리핑 (실거래) + 사전 컷 (deposit==0 + monthly_rent==0)
- **백분위 기준**: 서울 전체 425개 동 기준. 자치구 내 백분위 X
- **거래 3건 미만**: 같은 구 평균 → 서울 중위 fallback
- **지오코딩 단위**: 지번 단위만 (매물 단위 금지)
