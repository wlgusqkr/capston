## 매핑 요약 (실제 구현)

### 주요 변환

- `adong.adong_code` → `dong.code` (PK), `area_m2 / 1_000_000 → area_km2`, `boundary → geom`, `location → centroid`
- `dong.slug` 신규 생성: Django slugify(allow_unicode=True), 충돌 시 `-2`/`-3` 접미.
- `dong.gu` (string) ← Gu lookup으로 한글 구 이름.
- `dong.score_rent/amenity/transit = 0.0` (Phase 4에서 재계산).

### FK varchar id → bigint id 매핑

- `subway_congestion.station_id` (varchar) → SubwayStation lookup → bigint id
- `bus_congestion.bus_stop_id` (varchar) → BusStop lookup → bigint id
- 메모리 dict 한 번 적재 후 사용 (각각 400/15k entries).

### rent_deal dong 매핑 (PROTECT FK, NOT NULL)

2-pass 전략:
1. 1단계: `ldong → 대표 dong` fallback dict 메모리 생성 (ST_Intersects + 면적 max DESC). INSERT 시점에 fallback 사용.
2. 2단계: 모든 INSERT 후 `UPDATE rent_deal SET dong_id = (SELECT … ST_Contains(d.geom, r.geom))` 정밀 보정.

샘플(--limit 10000) 결과: 4,733/10,000 (47%)이 ST_Contains 보정으로 fallback과 다른 dong으로 정정됨. 이는 한 ldong이 여러 행정동에 걸칠 때 좌표 기반이 정답이라는 것을 확인.

### housing_type → deal_type 매핑

```python
HOUSING_TYPE_TO_DEAL_TYPE = {
    "아파트": "apt",
    "오피스텔": "officetel",
```

unknown은 `villa`로 폴백 (안전 디폴트, RDS 7가지 외 없음).

### bus_stop.adong_code 백필

- RDS 95% 매핑됨, 5% (719건)는 location도 NULL이라 ST_Contains 백필 불가능.
- 결과: 백필 후에도 NULL 719개 그대로 (모델 nullable로 완화하여 적재됨).
- **이 719건은 SubwayStation/BusStop처럼 좌표가 본질적으로 없는 데이터라 RDS의 한계**. 사용자에게는 NULL adong이 5%라고 안내.

## 멱등성 검증

- 모든 `INSERT … ON CONFLICT (pk_or_uq) DO UPDATE/NOTHING`.
- `--limit 10000` 샘플로 한 번 적재 후 전체 적재 시 ON CONFLICT가 동일 row를 업데이트만 (중복 행 0). bus_congestion / rent_deal 모두 검증.
- 파일/캐시 외부 의존 없음 — 어느 시점에 끊어도 다시 처음부터 돌릴 수 있음.

## 알려진 이슈

### 1. bus_stop.geom NULL 719건 (RDS 한계)
모델을 nullable로 완화. 백필 불가.

### 2. rent_deal에서 dong 매핑 fallback이 INSERT 시 사용된 후 ST_Contains UPDATE로 정밀 보정
- 만약 ST_Contains UPDATE가 끊어지면, 그때까지 dong이 fallback 그대로 남는 row가 있을 수 있음. 멱등 재실행 시 INSERT는 ON CONFLICT로 통과되지만 UPDATE는 별도 패스라 다시 돌려야 정확.
- 보정 SQL은 `WHERE r.dong_id <> sub.dong_id` 로 변경된 row만 업데이트하므로 두 번 돌려도 OK.

### 3. metric_code 30자 확장 (자동 적용된 마이그레이션)
RDS에 26자 코드(`POP_ELDERLY_RATIO_FORECAST` 등)가 있어 모델 max_length 20→30 변경. 0002_extend_metric_code_length.py 자동 생성, FK 컬럼들도 함께 변경됨.

### 4. RDS bus_congestion / rent_deal 큰 테이블 적재 시간
- 8M / 7.4M 행 INSERT executemany는 ~15분 / ~25분 소요.
- 더 빠르게 하려면 staging table COPY → INSERT ON CONFLICT 패턴 필요. 학부 프로젝트 1회성이라 단순화.

### 5. `Dong.score_*`은 모두 0.0
Phase 4에서 재계산 예정 (SPEC 11.2). 본 단계 범위 X.

### 6. `housing_type` "기타" 또는 unknown
RDS에 명시된 7가지 값(아파트/오피스텔/다세대/연립/연립다세대/다가구/단독) 외에는 없음을 검증. 만약 새로 들어오면 `villa`로 폴백.

### 7. 외부 5433 차단 운영 DB (slgi)
홈서버 이관 시 LOCAL_DSN을 GCP/홈서버 주소로 변경. 본 ETL은 그대로 재사용.

### 8. PostGIS 인덱스 사용 확인
- `dong.geom` GIST 인덱스 적용 (`dong_geom_f4fe7728_id`)
- `rent_deal.geom` GIST 인덱스 (`rentdeal_geom_gist_idx`, `rent_deal_geom_87de27b2_id`)
- ST_Contains 보정 쿼리는 두 GIST를 모두 사용해 최적화됨.
