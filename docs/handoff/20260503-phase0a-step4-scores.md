## 완료된 작업

### 1) `_compute_real()` 본문 작성
- `Dong` 425개 전체에 대해 `score_rent` / `score_amenity` / `score_transit` (각 0~100) 계산 후 `bulk_update`.
- raw metric 수집 → fallback 처리 → 백분위 변환 → 일괄 갱신 4단계.
- `composite` (총점)은 SPEC 11.1 따라 API 호출 시점에 가중치 곱해 계산 (저장 안 함).

### 2) Rent 점수 — 환산월세 백분위 (저렴=높음)
- **환산식**: `monthly_rent + max(0, deposit - 1000) / 100 * 0.5` (만원/월)
  - 보증금 1000만원 기준. 초과분 100만원당 0.5만원 월세 환산 (연 6% 전월세전환률 약식).
  - 전세 (`monthly_rent == 0`) 는 `deposit / 100 * 0.5` 로 전액 환산.
- **거래량 3건 미만 동 fallback** (SPEC 14.2):
  1. 같은 구의 ≥3건 충족 동 평균 (gu_avg)
  2. gu_avg 도 없으면 서울 전체 중위값
- 환산월세 작은 값(=저렴) → 백분위 → `100 - pct` (reverse=True)

### 3) Amenity 점수 — log-scale 가중합 백분위
- 카테고리별 카운트 → `log1p(count)` → 가중합 → 백분위.
- **가중치** (SPEC 11.2 미상세 — handoff Step 1b 권장 기준):

| 카테고리       | 가중치 |
|---|---|
| convenience    | 0.20  |
| hospital       | 0.15  |
| mart           | 0.10  |
| restaurant     | 0.10  |
| cafe           | 0.10  |
| studycafe      | 0.10  |
| pharmacy       | 0.10  |
| laundry        | 0.05  |
| oliveyoung     | 0.05  |
| park           | 0.05  |

- log-scale 도입 이유: `restaurant` 94,742건이 dominate하지 않게.

### 4) Transit 점수 — 지하철 + 버스 가중합 백분위
- **지하철** (60%): `max(0, 1 - dist_m / 1000)` — `NearestSubway.rank=1` 의 `distance_m` 사용. 1000m 초과는 0.
- **버스** (40%): `min(1, log1p(bus_count) / log1p(50))` — 50개 이상은 클램프 1.0.
- raw transit_metric → 백분위 (높을수록 높음).

### 5) 백분위 정규화
- 동률 처리: 같은 값에 평균 rank 부여 (예: fallback 47.8 그룹 322개 → 모두 동일 rank).
- N=425 기준 `(rank / (N-1)) * 100` → 0~100.
- None 값은 50.0 (사용 안 함 — fallback 으로 사전 채움).

## SPEC 11.2 해석에 추가한 가정 (handoff에 명시)

1. **환산월세 공식 (SPEC 11.2 미상세)**:
   - `monthly + max(0, deposit-1000)/100 * 0.5` 채택. 1000만원 초과분만 환산.
   - 다른 옵션: 모든 보증금을 월세 환산 (`monthly + deposit/100 * 0.5`) — 전세-월세 비교 시 후자가 더 공평하지만 보증금 분포가 매우 넓어(0~50,000) 학부 프로젝트엔 전자가 직관적.

2. **카테고리 가중치 (SPEC 11.2 미상세)**:
   - handoff Step 1b 권장값 그대로 채택. 실증적으로 자취 시그널과 부합 (편의점·병원 가중↑).
   - 추후 사용자 선호 학습으로 카테고리 가중도 학습 가능 (SPEC 11.4 확장 시).

3. **Transit 가중치**:
   - 지하철 60% / 버스 40%. SPEC은 "(가까운 역까지 거리 역수) + (버스 정류장 수)" 만 명시.
   - 거리 캡 1000m, 버스 캡 50개는 안전한 default. 양 끝단 클램프로 분포가 sane.

4. **Fallback 정책**:
   - SPEC 14.2는 "거래량 3건 미만 월은 점 생략" — 시계열용 가이드. 점수는 동 단위라 별도 처리.
   - "5개 구만 채우고 나머지는 null" 은 UI 구멍 발생 → 같은 구 평균 → 서울 중위 fallback 채택.
   - fallback 동은 score_rent 47.8 군집 (중앙 plateau). 시각적으로는 "rent 데이터 없음 = 중간색" 으로 보임. UI 측에서 rent 신뢰도 표시 추가 권장.

## 미완 / 알려진 이슈

### 1) Rent 데이터 spillover (Step 3 잔존 이슈)
- 성북구 보문동 5건이 고립적으로 fallback gu_avg 를 만듦 → 성북 19개 동이 score_rent=93.3 으로 동률.
- 향후 25개 구 풀 적재 시 자연 해소.

### 2) score_rent plateau
- 322개 동 (76%) 이 47.8 동률.
- 데모 시 "5개 구만 정확하고 나머지는 데이터 보강 중" 은 솔직히 안내 가능. UI 라벨 권장.

### 3) Transit 캡 (1000m, 50개) 의 임의성
- 현재 max 1.0 으로 클램프되는 dong 수: 지하철 0개 (모든 dong 의 거리가 1000m 미만 = 사실상 캡 없음), 버스 0개 (max 102지만 log scale 정규화 → 100/log1p(50) ≈ 1.0 정확히는 안 닿음).
- 즉 실 데이터 분포에서 cap 영향 미미. 분포가 자연스럽게 0~1 사이로 떨어짐.

### 4) 환산 계수 0.5 (월세전환률) 의 임의성
- 한국 표준 전월세전환률은 시기·지역마다 4~7% 사이. `deposit/100 * 0.5` 는 약 6% 환산 (1만원/월 = 보증금 200만원 가치).
- 절대값보다 상대 순위가 중요한 백분위 점수라 영향 미미.

### 5) Amenity 가중치 합 1.0 아님
- 합 = 1.00 (의도적). log1p(count) 의 절대 스케일은 백분위 변환에서 무관하므로 합 정규화는 불필요.
